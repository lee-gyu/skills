# 아이디어를 브레인스토밍하기

자연스러운 협업 대화를 통해 아이디어를 완성된 설계와 명세로 바꾸도록 돕습니다.

현재 프로젝트 맥락을 이해하는 것에서 시작한 뒤, 아이디어를 다듬기 위해 질문을 한 번에 하나씩 합니다.
무엇을 만들지 이해했다면 설계를 제시하고 사용자 승인을 받습니다.

## 체크리스트

아래 항목마다 작업을 만들고 순서대로 완료해야 합니다.

1. **프로젝트 맥락 탐색** - 파일, 문서, 최근 커밋 확인
2. **명확화 질문** - 한 번에 하나씩, 목적/제약/성공 기준 이해
3. **2-3가지 접근 제안** - trade-off와 추천안 포함
4. **설계 제시** - 복잡도에 맞춘 섹션으로 제시하고 각 섹션 뒤 사용자 승인 받기
5. **명세 자체 리뷰** - 플레이스홀더, 모순, 모호함, 범위에 대한 빠른 inline 확인(아래 참고)

## 프로세스

**아이디어 이해:**

- 먼저 현재 프로젝트 상태를 확인합니다(파일, 문서, 최근 커밋).
- 자세한 질문을 하기 전에 범위를 평가합니다. 요청이 여러 독립 하위 시스템을 설명한다면(예: "채팅, 파일 저장, 결제, 분석이 있는 플랫폼을 만들어줘"), 즉시 이를 지적하세요. 먼저 분해가 필요한 프로젝트의 세부 사항을 질문으로 다듬느라 시간을 쓰지 마세요.
- 프로젝트가 단일 spec으로 너무 크다면 사용자가 하위 프로젝트로 나누도록 돕습니다. 독립적인 조각은 무엇인지, 서로 어떻게 관련되는지, 어떤 순서로 만들어야 하는지 정합니다. 그런 다음 첫 번째 하위 프로젝트를 일반 설계 흐름으로 브레인스토밍합니다. 각 하위 프로젝트는 자체 spec -> plan -> implementation 사이클을 가집니다.
- 적절한 범위의 프로젝트라면 아이디어를 다듬기 위해 질문을 한 번에 하나씩 합니다.
- 가능하면 객관식 질문을 선호하되, 개방형도 괜찮습니다.
- 메시지 하나에 질문 하나만 합니다. 한 주제에 더 많은 탐색이 필요하면 여러 질문으로 나누세요.
- 목적, 제약, 성공 기준을 이해하는 데 집중합니다.

**접근 방식 탐색:**

- trade-off가 있는 2-3가지 다른 접근 방식을 제안합니다.
- 추천안과 이유를 포함해 대화체로 선택지를 제시합니다.
- 추천하는 선택지를 먼저 제시하고 그 이유를 설명합니다.

**설계 제시:**

- 무엇을 만들지 이해했다고 판단되면 설계를 제시합니다.
- 각 섹션은 복잡도에 맞춥니다. 단순하면 몇 문장, 미묘한 부분이 있으면 최대 200-300단어.
- 각 섹션 뒤 지금까지 맞아 보이는지 묻습니다.
- 아키텍처, 컴포넌트, 데이터 흐름, 오류 처리, 테스트를 다룹니다.
- 이해되지 않는 부분이 있으면 되돌아가 명확화할 준비를 합니다.

**격리와 명확성을 위한 설계:**

- 시스템을 더 작은 단위로 나눕니다. 각 단위는 하나의 명확한 목적을 갖고, 잘 정의된 인터페이스로 소통하며, 독립적으로 이해하고 테스트할 수 있어야 합니다.
- 각 단위에 대해 무엇을 하는지, 어떻게 사용하는지, 무엇에 의존하는지 답할 수 있어야 합니다.
- 내부를 읽지 않고도 단위가 하는 일을 이해할 수 있습니까? 소비자를 깨뜨리지 않고 내부를 바꿀 수 있습니까? 아니라면 경계를 다듬어야 합니다.
- 작고 경계가 잘 잡힌 단위는 당신이 작업하기도 더 쉽습니다. 한 번에 컨텍스트에 담을 수 있는 코드를 더 잘 추론하고, 파일이 집중되어 있을수록 편집이 더 신뢰할 수 있습니다. 파일이 커지면 대개 너무 많은 일을 하고 있다는 신호입니다.

**기존 코드베이스에서 작업:**

- 변경을 제안하기 전에 현재 구조를 탐색합니다. 기존 패턴을 따릅니다.
- 기존 코드에 작업에 영향을 주는 문제가 있다면(예: 너무 커진 파일, 불명확한 경계, 뒤엉킨 책임), 좋은 개발자가 작업 중인 코드를 개선하듯 설계에 타깃 개선을 포함합니다.
- 관련 없는 리팩터링을 제안하지 마세요. 현재 목표에 기여하는 것에 집중하세요.

## 핵심 원칙

- **한 번에 한 질문** - 여러 질문으로 압도하지 마세요.
- **가능하면 객관식 선호** - 가능할 때는 개방형보다 답하기 쉽습니다.
- **YAGNI를 엄격히 적용** - 모든 설계에서 불필요한 기능을 제거합니다.
- **대안 탐색** - 결정하기 전에 항상 2-3가지 접근을 제안합니다.
- **점진적 검증** - 설계를 제시하고, 넘어가기 전에 승인을 받습니다.
- **유연하게 대응** - 이해되지 않는 부분이 있으면 되돌아가 명확화합니다.
