# 에이전트 루프 엔지니어링

루프 자체의 제어 흐름 설계. 하네스가 "런타임"이라면 이 문서는 "제어 정책"이다.

## 종료 조건 (Stop conditions)

에이전트 루프의 종료는 세 종류다. 셋 다 명시적으로 설계하라:

1. **모델이 완료 선언**: 도구 호출 없이 텍스트로 응답. 기본 경로. 단, 모델은 조기 종료(덜 끝났는데 끝냈다고 함)하는 경향이 있다 — 시스템 프롬프트에 완료 기준("테스트가 통과할 때까지", "체크리스트 전 항목 확인")을 검증 가능하게 명시하면 크게 줄어든다.
2. **예산 소진**: 최대 턴 수, 최대 토큰(비용), 최대 시간(wall-clock). 셋 다 걸어라. 무한 루프는 코드 버그가 아니라 과금 사고로 나타난다. 예산 소진 시 그냥 죽이지 말고, 마지막 턴에 "예산이 다 됐다. 현재까지의 상태와 남은 일을 정리하라"는 기회를 주면 재개 가능한 형태로 끝난다.
3. **정책적 중단**: 반복 감지(같은 도구를 같은 인자로 N번 — 모델이 루프에 빠진 신호), 위험 행동 감지, 사용자 취소.

## 실패 처리와 복구

- **도구 실패는 루프의 정상 경로다.** 예외가 아니라 데이터로 취급하라. 실패 결과를 모델에게 돌려주면 대부분 스스로 복구한다(재시도, 다른 방법 시도). 하네스가 자동 재시도하는 것은 일시적 오류(네트워크, 429)에만 — 논리적 오류(잘못된 인자)는 모델이 봐야 고친다.
- **연속 실패 카운터**: 같은 도구가 연속 N번 실패하면 개입하라. 모델에게 "이 접근은 작동하지 않는다. 전략을 바꿔라"를 주입하거나, 사람에게 에스컬레이션. 모델은 실패한 접근을 고집하는 경향이 있다(sunk cost가 컨텍스트에 쌓여 있어서).
- **컨텍스트 오염 후 복구**: 잘못된 방향으로 한참 간 경우, 그 시행착오 이력 자체가 계속 모델을 오염시킨다. 이력을 계속 끌고 가는 것보다, 배운 것만 요약하고 컨텍스트를 리셋해서 재시작하는 것이 이기는 경우가 많다. "git reset하고 다시 짜는 게 빠른" 상황과 같다.

## Human-in-the-loop 게이트

- 행동을 두 부류로 나눠라: **되돌릴 수 있는 것**(파일 편집, 조회, 브랜치에 커밋)과 **되돌릴 수 없는 것**(배포, 이메일 발송, 결제, 삭제, main push). 전자는 자율 실행, 후자는 승인 게이트.
- 승인 게이트의 UX: "무엇을, 왜" 하려는지 모델의 의도와 함께 제시. 승인 요청이 너무 잦으면 사람이 무지성 승인하게 되어(consent fatigue) 게이트가 무력화된다 — 게이트는 정말 위험한 것에만 아껴 써라.
- 자율성 다이얼: 신뢰가 쌓이기 전에는 read-only → 승인 하 쓰기 → 자율 쓰기 순으로 점진적으로 올려라. 처음부터 풀 자율은 신뢰 사고 한 번에 프로젝트가 죽는다.

## 피드백 루프의 질이 에이전트의 상한이다

- 에이전트가 잘 작동하는 도메인의 공통점: **저렴하고 빠른 검증 수단**이 있다. 코딩(테스트, 컴파일러, 린터), 데이터 분석(실행 결과). 검증 수단이 없는 도메인(전략 수립, 글쓰기)에서는 에이전트가 자기 오류를 감지 못 하고 자신 있게 표류한다.
- 설계 함의: 도구 목록에 "검증 도구"를 반드시 포함시켜라. 테스트 실행, 스키마 검증, 스크린샷 확인. 그리고 시스템 프롬프트에 "행동 후 검증"을 루틴으로 박아라. 검증 없는 에이전트 루프는 어서션 없는 테스트 스위트다 — 도는 것 같지만 아무것도 보장하지 않는다.

## 루프 구조의 변형들

- **Plan-then-execute**: 실행 전에 계획을 먼저 세우게 하고(별도 턴 또는 문서로), 계획을 사람이 승인 후 실행. 긴 태스크에서 방향 오류의 비용을 크게 줄인다. 계획은 파일로 남겨 컴팩션에도 살아남게.
- **Reflection 턴**: 주기적으로(예: 10턴마다) "지금까지 진행을 점검하고 원래 목표와 어긋난 게 없는지 확인하라"는 턴을 강제 주입. 장기 태스크의 표류(drift) 방지.
- Opinionated: ReAct니 Tree-of-Thoughts니 하는 논문 패턴을 그대로 구현하려 하지 마라. 현대 모델은 자체 추론(extended thinking)이 내장되어 있어서, 구식 프롬프트 패턴을 덧씌우면 오히려 방해된다. 루프 설계는 위의 실무 관심사(종료, 예산, 검증, 게이트)에 집중하는 게 낫다.
