# 컨텍스트 엔지니어링 (+ 메모리/컴팩션)

## 핵심 원칙

**컨텍스트는 유한하고, 채울수록 성능이 떨어진다.** "컨텍스트 윈도우가 200K니까 넉넉하다"는 착각이다. 두 가지 이유:

1. **Context rot**: 컨텍스트가 길어질수록 모델의 주의(attention)가 희석되어 중간에 있는 정보를 놓치거나 오래된 정보에 끌려간다. 토큰 예산은 RAM 용량이라기보다 "인지 예산"이다.
2. **비용과 지연**: 매 턴 전체 컨텍스트를 다시 처리한다(프롬프트 캐싱이 완화하지만 무료는 아니다).

목표 함수: **원하는 결과의 확률을 최대화하는 최소한의 고신호(high-signal) 토큰 집합을 찾아라.** 이것이 컨텍스트 엔지니어링의 전부다. 프롬프트 엔지니어링은 "무엇을 말할까"였고, 컨텍스트 엔지니어링은 "매 턴, 무엇을 컨텍스트에 남기고 뺄까"라는 상태 관리 문제다.

## 컨텍스트의 구성과 각각의 원칙

### 시스템 프롬프트
- "적정 고도"를 찾아라. 너무 구체적(하드코딩된 if/else 지시 뭉치)이면 취약하고 유지보수 지옥이 된다. 너무 추상적("좋은 어시스턴트가 되어라")이면 신호가 없다. 휴리스틱과 원칙을 주되, 모델이 판단할 여지를 남겨라.
- 구조화(섹션 구분, XML 태그나 마크다운 헤더)는 도움이 된다. 하지만 프롬프트에 규칙을 계속 덧대는 것은 코드에 특수 케이스 if를 덧대는 것과 같은 냄새다. 규칙이 계속 늘어난다면 도구/구조 설계가 잘못됐다는 신호일 수 있다.

### 도구 결과 (컨텍스트의 최대 소비자)
- 대부분의 에이전트에서 컨텍스트의 70~90%는 도구 결과다. 여기가 최적화 1순위.
- 도구가 "필요한 정보만, 모델이 읽기 좋은 형태로" 반환하게 하라. DB 조회 도구가 raw row 전체를 JSON으로 덤프하면 안 된다. `verbose` 플래그를 두고 기본은 요약 반환하는 패턴이 좋다.
- 오래된 도구 결과는 가치가 급감한다. 3턴 전의 파일 내용 전문이 아직 필요할 확률은 낮다 — 컴팩션 대상 1순위(아래 참조).

### Retrieval: pre-load vs just-in-time
- 임베딩 기반 RAG로 미리 다 넣어주는 방식 vs 모델에게 검색 도구(grep, 파일 읽기, 시맨틱 검색)를 주고 필요할 때 직접 찾게 하는 방식.
- **Opinionated: 에이전트에는 just-in-time이 기본값이다.** 이유: (1) 미리 넣은 컨텍스트는 대부분 안 쓰이고 자리만 차지한다, (2) 모델이 탐색하며 점진적으로 이해를 쌓는 것이 정확도가 높다(사람이 코드베이스를 파악하는 방식과 같다), (3) 인덱스 신선도 문제가 없다. 비용은 지연시간 — 지연이 극단적으로 중요하고 코퍼스가 안정적이면 하이브리드(핵심은 pre-load, 나머지는 검색)로.
- 파일시스템 자체가 최고의 컨텍스트 관리 도구다. 경로, 디렉토리 구조, 파일명이 메타데이터 역할을 하고, 모델은 필요한 부분만 읽는다.

## 장기 실행: 컨텍스트가 넘칠 때의 세 가지 기법

한 번의 태스크가 컨텍스트 윈도우보다 커질 때. 우선순위 순:

### 1. 컴팩션 (Compaction)
- 컨텍스트가 임계치(예: 70~80%)에 도달하면, 지금까지의 대화를 요약해서 새 컨텍스트로 시작한다. 요약에 반드시 보존할 것: 원래 목표, 지금까지의 핵심 결정과 이유, 현재 상태, 남은 일, 실패했던 시도(같은 실수 반복 방지). 버릴 것: 오래된 도구 결과 원문, 시행착오의 세부 과정.
- 더 가벼운 버전: 전체 요약 전에, 오래된 도구 결과만 placeholder("[결과 생략 — 필요하면 다시 호출]")로 치환하는 것만으로 큰 효과. 손실이 적어 먼저 시도할 것.
- 주의: 컴팩션은 손실 압축이다. 공격적으로 하면 모델이 "왜 이 결정을 했는지"를 잊고 했던 실수를 반복한다. 요약 프롬프트 자체를 실제 실패 사례로 튜닝해야 한다.

### 2. 구조화된 노트 (Structured note-taking / 외부 메모리)
- 모델이 컨텍스트 밖(파일)에 작업 노트를 쓰게 하라: TODO 목록, 진행 상황, 발견한 제약 조건. 컴팩션이 일어나도 파일은 남고, 모델은 다시 읽으면 된다.
- Claude Code의 TODO 리스트, `NOTES.md` 패턴이 이것이다. "컨텍스트는 캐시, 파일시스템이 영속 저장소"라는 계층 구조.

### 3. 서브에이전트로 컨텍스트 격리
- 탐색량이 큰 서브태스크(예: "이 코드베이스에서 인증 흐름 파악")를 서브에이전트에 위임하고, 요약만 받아온다. 서브에이전트가 수만 토큰을 태워도 부모 컨텍스트에는 정제된 결과만 들어온다. `multi-agent.md` 참조.

## 세션 간 메모리 (cross-session memory)

- 세션 내 메모리(위)와 구분하라. 세션 간 메모리는 "사용자/프로젝트에 대해 학습한 것을 다음 세션에 가져가는 것"이다.
- Opinionated: 벡터DB에 대화를 다 넣는 방식보다, **모델이 명시적으로 큐레이션한 소량의 텍스트 파일**(예: 프로젝트별 CLAUDE.md, 사용자 선호 메모)이 훨씬 잘 작동한다. 자동 수집된 메모리는 노이즈가 신호를 압도하고, 낡은 정보가 오답을 유발한다. 메모리는 쌓는 것보다 지우는 것이 어렵다 — 처음부터 "무엇을 기억할 가치가 있는가"를 보수적으로.

## 안티패턴 체크리스트

- 시스템 프롬프트에 "혹시 몰라서" 넣은 문서 전문 → 빼고 검색 도구로.
- 도구 20개를 전부 등록 → 태스크에 필요한 것만. 유사한 도구가 많으면 모델이 선택을 헷갈린다.
- few-shot 예제 잔뜩 → 에이전트에서는 다양하고 정제된 소수 예제가 낫다. 예제 뭉치는 컨텍스트만 먹는다.
- 대화 이력을 무한히 append → 컴팩션 전략 없는 에이전트는 장기 태스크에서 반드시 무너진다.
