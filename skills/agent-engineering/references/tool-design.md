# 도구(Tool) 설계

도구는 모델을 위한 API다. 소비자가 "코드를 읽을 수 없고 문서만 읽는, 매우 똑똑한 신입"이라고 생각하라. API 설계 경험이 그대로 이식되는 영역이지만, 사람용 API와 결정적으로 다른 점들이 있다.

## 사람용 API와 다른 점

1. **문서가 곧 인터페이스다.** 사람은 문서가 부실해도 소스를 까보거나 시행착오로 배우지만, 모델은 매 호출을 description만 보고 결정한다. description에 언제 쓰는지/언제 쓰면 안 되는지/인자 형식/반환 형태/예시를 다 담아라. 팀의 신입에게 설명하듯 써라. 모호한 설명은 곧바로 오호출로 나타난다.
2. **응답도 컨텍스트 비용이다.** 사람용 API는 완전한 데이터를 주고 클라이언트가 골라 쓰지만, 도구 응답은 전부 컨텍스트에 들어가 이후 모든 턴의 비용과 노이즈가 된다. 기본은 요약/핵심만, 상세는 opt-in(`detail: "full"` 파라미터)으로.
3. **선택지가 많으면 성능이 떨어진다.** 사람용 SDK는 메서드 100개여도 되지만, 유사한 도구가 많으면 모델이 선택을 헷갈린다. 도구 수를 늘리기 전에 "기존 도구의 파라미터로 흡수할 수 있는가"를 먼저 물어라.

## Opinionated 원칙들

### 소수의 고레버리지 도구
- 내부 API 엔드포인트를 1:1로 도구화하지 마라(흔한 실수). 모델의 *워크플로우* 단위로 도구를 설계하라. 예: `list_users` + `get_user` + `get_user_orders`를 각각 노출하는 대신 `search_users(query, include: ["orders"])` 하나가 낫다. 여러 호출을 하나로 합치면 턴 수와 컨텍스트 소비가 같이 줄어든다.
- 판단 기준: "이 태스크를 사람 인턴에게 시킨다면 어떤 화면/명령이 필요할까?" — CRUD 엔드포인트 목록이 아니라 그 답이 도구 목록이다.

### 범용 도구가 특수 도구를 이긴다 (특히 코딩 에이전트)
- bash + 파일 읽기/쓰기/편집 + grep이면 모델이 조합해서 거의 뭐든 한다. 특수 도구는 (1) 자주 쓰이고, (2) 범용 도구 조합이 비싸거나 위험할 때만 추가. 도구 추가는 공짜가 아니다 — 스키마가 매 호출의 컨텍스트를 차지한다.

### 반환 형식
- 모델이 읽을 것이므로 사람이 읽기 좋은 형식이 대체로 모델에게도 좋다. 깊게 중첩된 JSON보다 마크다운/평문 요약이 나은 경우가 많다. 단, 모델이 이후 도구 호출의 인자로 그대로 써야 하는 값(ID 등)은 정확한 형태로 명시적으로 반환하라.
- 자연스러운 다음 행동을 안내하라: 검색 결과가 0건이면 빈 배열만 주지 말고 "0건. 더 짧은 키워드로 시도해보라"를, 페이지네이션이면 "다음 페이지를 보려면 cursor=X"를 함께.

### 에러 반환
- 에러 메시지는 모델의 복구 능력을 결정한다. 나쁨: `Error: 400`. 좋음: `"date는 YYYY-MM-DD 형식이어야 합니다. 받은 값: '3월 5일'"`. 유효성 검사는 스키마(required, enum)로 최대한 앞당기고, 런타임 에러는 교정 방법을 담아 반환하라.

### 위험한 도구
- 파괴적 연산은 별도 도구로 분리하고(`delete_records`를 `search`의 플래그로 만들지 마라), description에 위험성을 명시하고, 하네스 레벨에서 승인 게이트를 걸어라. 방어를 모델의 판단에만 맡기지 마라 — `agent-loop.md`의 human-in-the-loop 참조.
- 멱등성을 설계하라. 모델은 타임아웃 후 재시도한다. 결제/발송류 도구에 idempotency key가 없으면 이중 실행 사고가 난다. 웹훅 설계와 같은 감각.

## 스키마 작성 팁

- 파라미터 이름은 자기설명적으로 (`user` 대신 `user_id`, `ts` 대신 `created_after_iso8601`).
- enum과 required를 적극 사용 — 모델의 자유도를 줄이는 것이 정확도를 올린다.
- 파라미터가 5개를 넘어가면 도구가 두 개여야 한다는 신호일 수 있다.
- description에 좋은 호출 예시 1~2개를 포함하면 오호출이 눈에 띄게 준다.

## MCP (Model Context Protocol)

- 도구를 노출하는 표준 프로토콜. "도구계의 REST/OpenAPI"로 이해하면 된다. 서버(도구 제공자)와 클라이언트(하네스)를 분리해서, 한 번 만든 도구 서버를 Claude Code, Claude.ai, 자체 하네스 어디서든 재사용.
- 언제 MCP로: 도구를 여러 하네스/팀에서 공유할 때, 서드파티 통합을 쓸 때. 단일 애플리케이션 내부 전용 도구라면 그냥 인프로세스 함수로 충분하다 — 프로토콜 오버헤드를 정당화할 이유가 없다.
- 주의: 서드파티 MCP 서버는 공급망 리스크다(도구 결과를 통한 프롬프트 인젝션 벡터). npm 패키지 고르듯 검증하라.

## 도구 개선 루프

도구가 잘 안 쓰이거나 오호출될 때: 실제 트랜스크립트를 읽어라. 모델이 어떤 description을 보고 왜 그 선택을 했는지 추적하면 대부분 description의 모호함이나 유사 도구 간 경계 불명이 원인이다. 도구 정의를 고치고 같은 태스크를 재실행해 비교하라 — 이것이 도구의 단위 테스트다.
