# 평가(Evals)와 프로덕션 운영

## 평가: 에이전트의 테스트 스위트

**evals 없는 에이전트 개발은 테스트 없는 리팩터링이다.** 프롬프트나 도구를 고치면 어떤 케이스는 좋아지고 다른 케이스는 조용히 나빠진다. 회귀를 감지할 수단이 없으면 개선은 체감과 미신으로 퇴화한다.

### Opinionated: evals를 시작하는 올바른 순서

1. **실패 사례에서 시작하라, 벤치마크에서 시작하지 마라.** 공개 벤치마크 점수는 여러분의 태스크와 상관이 낮다. 실제 사용에서 실패한 케이스 5~10개를 모아 그것이 첫 eval set이다. 버그 리포트가 회귀 테스트가 되는 것과 같은 흐름.
2. **작게 시작해도 된다.** 20개짜리 eval set도 없는 것보다 압도적으로 낫다. "통계적으로 유의미한 규모가 될 때까지 안 만든다"는 완벽주의가 최악의 선택.
3. **변경마다 돌려라.** 프롬프트 수정, 도구 description 수정, 모델 교체 — 전부 eval을 통과해야 머지. CI에 넣어라.

### 무엇을 어떻게 채점하나

에이전트 평가의 특수성: 출력이 비결정적이고, 같은 결과에 도달하는 경로가 여러 개다. 채점 방법을 계층화하라:

- **코드 기반 어서션 (가장 신뢰, 가능하면 항상)**: 최종 *상태*를 검증하라, 경로가 아니라. "테스트가 통과하는가", "파일이 올바른 내용으로 존재하는가", "DB 레코드가 갱신됐는가". 도구 호출 순서를 어서션하는 것은 취약하다 — 모델이 다른 유효한 경로를 찾으면 깨진다. (E2E 테스트에서 DOM 구조가 아니라 사용자 가시 결과를 검증하라는 원칙과 같다.)
- **LLM-as-judge (주관적 품질)**: 문서 품질, 답변 정확성 등. 반드시 루브릭(구체적 채점 기준)을 주고, 판정 결과 샘플을 사람이 주기적으로 검수해서 judge 자체를 캘리브레이션하라. 캘리브레이션 안 된 judge는 노이즈 발생기다.
- **인간 검수**: judge의 정답지 구축용 + 정기 샘플링. 전량 검수는 확장 불가능하므로 자동화의 감사(audit) 역할로.

또한 pass/fail만 보지 말고 **비용과 턴 수를 함께 기록하라**. 정확도가 같은데 토큰을 3배 쓰는 회귀는 pass/fail 지표에 안 잡힌다.

### 흔한 함정
- eval set 과적합: 20개를 계속 고치다 보면 그 20개에만 좋은 프롬프트가 된다. 주기적으로 신규 실패 사례를 추가하고, 일부는 held-out으로 남겨라.
- 비결정성 무시: 한 번 돌려 통과/실패를 판정하지 마라. 흔들리는 케이스는 N회 실행해 통과율로 봐라(flaky test 다루는 감각).

## 관측성 (Observability)

- **트랜스크립트가 근본 데이터다.** 요청/응답 로그 수준이 아니라, 매 세션의 전체 메시지 배열(시스템 프롬프트 버전, 도구 호출과 결과, 모델 응답 전문)을 저장하라. 에이전트 디버깅의 90%는 "트랜스크립트를 읽으면" 풀린다 — 모델이 왜 그랬는지는 모델이 본 것을 봐야 안다.
- 구조화 이벤트로도 남겨라: 세션 ID, 턴 번호, 도구명, 지연시간, 토큰 수, 에러, 종료 사유. 대시보드와 알림은 이걸로 만든다.
- 추적할 핵심 지표: 태스크 성공률(정의 필요), 세션당 비용 분포(평균이 아니라 p95/p99 — 폭주 세션이 꼬리에 있다), 턴 수 분포, 도구별 실패율, 예산 소진으로 죽은 세션 비율.
- 개인정보: 트랜스크립트에는 사용자 데이터가 통째로 들어간다. 보존 기간, 접근 통제, 마스킹 정책을 처음부터 설계하라. "일단 다 로깅"은 나중에 컴플라이언스 폭탄이 된다.

## 비용 통제

- 에이전트 비용은 분포의 꼬리가 지배한다. 정상 세션 $0.10, 폭주 세션 $30 같은 구조. 방어선: (1) 세션당 하드 토큰/비용 캡, (2) 반복 행동 감지 후 중단, (3) 사용자/테넌트별 rate limit + 일일 예산.
- **프롬프트 캐싱은 에이전트의 필수 최적화다.** 매 턴 시스템 프롬프트 + 누적 이력이 재전송되는 구조라, 캐싱 유무로 비용이 크게 달라진다(캐시 히트 입력은 원가의 소수 분의 일). 캐시가 깨지지 않게 프리픽스를 안정적으로 유지하라 — 시스템 프롬프트에 현재 시각 같은 가변 값을 넣으면 매 턴 캐시 미스가 난다.
- 모델 라우팅(쉬운 태스크는 작은 모델)은 유효하지만, evals가 갖춰진 뒤에 하라. 측정 없이 다운그레이드하면 "싸졌는데 왜 나빠졌는지"를 알 수 없다.

## 배포와 릴리스

- 프롬프트와 도구 정의는 코드다: 버전 관리, 코드 리뷰, 롤백 대상. 트랜스크립트 로그에 프롬프트 버전을 남겨야 "이 회귀가 어느 배포부터인지"를 추적할 수 있다.
- 점진적 릴리스가 특히 중요하다: 에이전트 변경의 영향은 비결정성 때문에 스테이징에서 다 안 보인다. 카나리(트래픽 일부에 신버전) + 지표 비교 후 확대.
- 모델 버전 고정(pinning): 모델 업그레이드는 의존성 메이저 버전 업그레이드로 취급하라. eval 전체 통과 확인 후 교체. "자동으로 최신 모델"은 조용한 대규모 회귀의 레시피다.
