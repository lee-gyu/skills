# 에이전트의 구성 요소

## 정의부터 정확히

업계에서 "에이전트"는 마케팅 용어로 오염됐다. 엔지니어링 관점의 정의:

- **워크플로우(Workflow)**: LLM 호출들이 *코드가 정의한 경로*를 따라 조합된 것. 분기, 체이닝, 병렬화가 있어도 제어 흐름은 개발자가 소유한다.
- **에이전트(Agent)**: LLM이 *스스로 제어 흐름을 결정*하는 것. 어떤 도구를 몇 번 호출할지, 언제 끝낼지를 모델이 정한다.

웹 개발 유비: 워크플로우는 명시적 라우팅이 있는 컨트롤러, 에이전트는 런타임에 실행 계획을 스스로 세우는 쿼리 플래너에 가깝다. 쿼리 플래너가 편리하지만 가끔 이상한 실행 계획을 짜듯, 에이전트도 그렇다.

## Opinionated: 복잡도 사다리

**아래 단계에서 해결되면 위로 올라가지 마라.** 각 단계는 비용, 지연시간, 비결정성이 증가한다.

1. **단일 LLM 호출** + 좋은 프롬프트 + (필요하면) retrieval. 놀랄 만큼 많은 문제가 여기서 끝난다.
2. **워크플로우 패턴들**:
   - *Prompt chaining*: 단계 분해. 각 단계 출력을 코드로 검증 가능(게이트).
   - *Routing*: 분류 후 특화된 프롬프트로 분기. if/else의 LLM 버전.
   - *Parallelization*: 독립 서브태스크 동시 실행, 또는 같은 태스크 N번 돌려 voting.
   - *Orchestrator-workers*: LLM이 서브태스크를 동적으로 나누고 워커에게 위임.
   - *Evaluator-optimizer*: 생성 → LLM 평가 → 피드백 반영 재생성 루프.
3. **에이전트**: 경로를 미리 정의할 수 없을 때만. 판별 기준 — "이 문제를 푸는 단계를 flowchart로 그릴 수 있는가?" 그릴 수 있으면 워크플로우로 만들어라. 더 싸고, 빠르고, 디버깅 가능하다.

에이전트가 정당화되는 신호: 필요한 스텝 수를 예측할 수 없음, 중간 결과에 따라 전략이 바뀜, 탐색적 성격(코딩, 리서치, 트러블슈팅).

## 네 가지 구성 요소

### 1. 모델
- 고르는 것이지 만드는 것이 아니다. 선택 기준: 도구 호출 신뢰도 > 벤치마크 점수. 에이전트에서는 단일 응답 품질보다 "여러 턴에 걸쳐 일관되게 도구를 잘 쓰는가"가 지배적이다.
- 비용 최적화는 라우팅으로: 쉬운 턴은 작은 모델, 어려운 계획 수립은 큰 모델. 단, 조기 최적화하지 마라 — 일단 가장 좋은 모델로 동작시킨 뒤 다운그레이드 실험을 한다. 반대 방향(작은 모델로 시작해서 안 되면 키우기)은 "안 되는 이유가 모델 탓인지 설계 탓인지" 구분을 불가능하게 만든다.

### 2. 도구
- 모델의 손발. 별도 문서(`tool-design.md`) 참조. 핵심만: 도구는 모델이라는 "매우 똑똑하지만 문서만 읽고 처음 쓰는 신입"을 위한 API다. 문서(description)의 품질이 곧 도구의 품질.

### 3. 컨텍스트
- 모델의 유일한 작업 메모리. 시스템 프롬프트 + 대화 이력 + 도구 결과 전부. 별도 문서(`context-engineering.md`) 참조.

### 4. 하네스
- 루프를 돌리는 런타임 전체. 별도 문서(`harness.md`) 참조. 모델 능력이 같아도 하네스 품질에 따라 에이전트 성능이 크게 갈린다 — 같은 엔진을 얹어도 섀시에 따라 차가 달라지는 것과 같다.

## 프레임워크에 대한 입장

**시작은 직접 API 호출로.** LangChain/LangGraph 류의 프레임워크에 대한 판단 기준:

- 에이전트 루프의 본질은 위 30줄짜리 while 루프다. 이걸 감추는 추상화는 학습 단계에서 해롭다. 프롬프트가 실제로 어떻게 조립되는지, 도구 결과가 어떤 형태로 들어가는지 모르면 성능 문제를 디버깅할 수 없다 — ORM만 쓰다 슬로 쿼리를 만난 상황과 같다.
- 프레임워크가 정당화되는 경우: 팀이 이미 본질을 이해하고 있고, 상태 영속화/재시도/휴먼 승인 같은 배관 코드를 재사용하려는 경우. 이때도 "추상화 아래에서 무슨 프롬프트가 나가는지 덤프해볼 수 있는가"를 도입 기준으로 삼아라.
- MCP(Model Context Protocol)는 프레임워크가 아니라 도구 연결 표준이다. 도구 서버 생태계 재사용이 목적이면 이쪽이 먼저다.

## 흔한 오해 교정

- "에이전트는 자율적일수록 좋다" → 아니다. 자율성은 신뢰도와 트레이드오프. 검증 가능한 피드백 루프(테스트, 컴파일러, 린터)가 있는 도메인(코딩)에서 자율성이 잘 작동하고, 피드백이 없는 도메인에서는 폭주한다.
- "프롬프트 엔지니어링이 핵심이다" → 단일 호출 시대의 관점. 에이전트에서는 컨텍스트 엔지니어링(무엇이 컨텍스트에 들어가고 남는가)과 도구 설계가 지배적이다.
- "멀티에이전트가 더 강력하다" → 대부분의 경우 단일 에이전트 + 좋은 도구가 이긴다. `multi-agent.md` 참조.
