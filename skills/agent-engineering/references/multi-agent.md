# 서브에이전트와 멀티에이전트

## 냉정한 출발점

**기본값은 단일 에이전트다.** 멀티에이전트는 분산 시스템이다 — 웹 개발자라면 이 문장만으로 경계심이 들어야 한다. 모놀리스로 충분한데 마이크로서비스를 쪼개면 얻는 것(조직적 확장성)은 없고 비용(조정, 정합성, 디버깅 난이도)만 남는 것과 정확히 같은 실패 양상이 나온다. "에이전트별로 페르소나를 나눠서 협업시키면 잘하겠지"는 대부분 환상이다 — 같은 모델을 여러 번 부르는 것이지 팀이 생기는 게 아니다.

멀티에이전트가 정당화되는 조건은 사실상 하나로 수렴한다: **컨텍스트 분리가 필요할 때.**

## 정당화되는 두 가지 패턴

### 1. 컨텍스트 격리형 서브에이전트 (가장 흔하고 유용)
- 문제: 탐색량이 큰 서브태스크(대규모 코드베이스 조사, 문서 더미 검색)가 부모의 컨텍스트를 오염시킨다.
- 해법: 서브에이전트를 스폰해서 위임. 서브에이전트는 자기 컨텍스트에서 수만 토큰을 태우며 탐색하고, **정제된 요약만** 부모에게 반환한다. 부모 컨텍스트에는 결과 요약 몇백 토큰만 남는다.
- 유비: 워커 스레드/잡 큐. 무거운 작업을 오프로드하고 결과만 받는다. 서브에이전트는 "동료"가 아니라 "일회용 워커 프로세스"다.
- 구현: 부모의 도구 목록에 `spawn_subagent(task, tools?)` 도구를 추가하는 것뿐이다. 특별한 프레임워크가 필요 없다.

### 2. Orchestrator-Workers (병렬 리서치형)
- 문제: 독립적으로 병렬화 가능한 넓은 탐색(예: "경쟁사 10곳 조사").
- 해법: 리드 에이전트가 태스크를 분해해 워커 N개에 동시 위임, 결과를 취합. 순차 대비 지연시간이 크게 줄어든다.
- 잘 맞는 조건: **읽기 중심 + 서브태스크 간 독립**. 리서치가 전형. 반대로 쓰기 중심 + 상호의존적 태스크(여러 에이전트가 한 코드베이스를 동시 수정)는 최악의 조합 — 충돌과 정합성 문제가 사람의 머지 컨플릭트보다 훨씬 심하게 난다.
- 비용 경고: 멀티에이전트 리서치는 단일 채팅 대비 토큰을 한 자릿수 배(수배~15배가량) 더 태운다. 결과 가치가 토큰 비용을 정당화하는 고가치 태스크에만 써라.

## 설계 시 실무 포인트

### 위임 인터페이스가 전부다
- 실패의 대부분은 "리드가 워커에게 태스크를 모호하게 설명"해서 생긴다. 워커는 부모의 컨텍스트를 못 본다(그게 목적이니까). 위임 프롬프트에 명시하라: 목표, 출력 형식, 범위/경계(무엇은 하지 마라), 사용할 도구, 노력 예산(가벼운 조사인지 심층 조사인지).
- 이건 정확히 "주니어에게 티켓 쓰는 법" 문제다. 재현 조건 없는 버그 티켓을 받은 주니어가 헤매듯, 모호한 위임을 받은 워커는 중복 작업을 하거나 빗나간다.

### 결과 취합
- 워커 반환 형식을 구조화하라(발견 사항, 근거/출처, 확신도, 못 찾은 것). 리드가 자유 텍스트 N개를 취합하는 것보다 훨씬 안정적이다.
- 워커의 실패(빈 결과, 에러)를 리드가 인지하고 재위임할 수 있게 하라. 조용히 빠진 결과는 취합 단계에서 "조사했는데 없음"으로 둔갑한다.

### 공유 상태
- 에이전트 간 통신은 메시지 전달(위임/반환)이 기본. 그 이상의 협업이 필요하면 파일시스템을 공유 상태로 쓰는 것이 실용적이다(워커가 파일로 산출물을 남기고 리드가 읽음). 에이전트 간 실시간 양방향 대화 설계는 복잡도 대비 이득이 거의 없다 — 필요하다고 느껴지면 태스크 분해가 잘못된 것이다.

### 디버깅과 관측성
- 멀티에이전트의 트랜스크립트는 트리 구조가 된다. 분산 트레이싱과 동일하게 접근하라: 위임마다 trace/span ID를 부여하고, 부모-자식 관계를 로그에 남겨라. 이게 없으면 "왜 이런 결과가 나왔는지"를 재구성할 수 없다.

## 안티패턴

- 역할극 멀티에이전트("PM 에이전트, 개발자 에이전트, QA 에이전트가 회의"): 토큰만 태우는 연극. 컨텍스트 분리 목적이 없다면 단일 에이전트의 단일 컨텍스트에서 순차로 하는 게 낫다.
- 워커에게 부모와 동일한 풀 도구셋 부여: 워커의 도구는 태스크에 필요한 최소로. 권한 최소화 원칙은 여기서도 유효하다.
- 깊은 중첩(서브에이전트의 서브에이전트의…): 2단계를 넘는 위임은 거의 항상 설계 신호가 아니라 통제 상실 신호다.
